# 家庭财务小管家需求规格说明书

版本：V1.1（改进版）  
日期：2026-04-02  
状态：评审稿

## 1. 引言

### 1.1 编写目的
本文档用于定义“家庭财务小管家”下一阶段改进需求，作为产品、开发、测试和验收的统一依据，确保后续迭代具备明确范围、可实现性与可验证性。

### 1.2 项目背景
当前系统已支持手工记账、OCR 图片导入、支付宝 CSV 导入、微信 XLSX 导入、分类统计与月度趋势展示，能够满足基础记账需求。  
随着记录量增加与家庭协同需求上升，现阶段存在以下改进诉求：
1. 记录检索能力不足，缺少多条件筛选与排序。
2. 记录维护能力有限，缺少“编辑”能力。
3. 缺少预算管理与超支提醒。
4. 缺少显式备份与恢复能力。
5. 导入质量与异常提示需要进一步标准化。

### 1.3 定义与缩写
1. SRS：Software Requirements Specification，需求规格说明书。
2. OCR：Optical Character Recognition，光学字符识别。
3. 幂等导入：重复导入同一账单文件不会产生重复记录。
4. P0/P1/P2：需求优先级。P0 为本期必做，P1 为本期应做，P2 为可延期。

## 2. 总体描述

### 2.1 产品视角
本系统是一个本地优先的家庭财务管理工具，采用前后端分离架构。
1. 前端：浏览器端页面，负责表单录入、导入预览、图表展示、交互反馈。
2. 后端：提供交易、统计、导入、系统信息接口。
3. 数据层：SQLite 本地数据库，保存交易数据和导入去重关键字段。

### 2.2 用户类与特征
1. 家庭记账用户（主要角色）：每日录入、导入、查询和查看统计。
2. 家庭管理员（可选角色）：维护分类、执行备份恢复、管理预算阈值。
3. 维护人员（开发/测试）：验证接口、排查导入异常、回归测试。

### 2.3 运行环境
1. 操作系统：Windows 优先，兼容 macOS/Linux 开发环境。
2. 后端：Python 3.10+，FastAPI，SQLAlchemy，SQLite。
3. 前端：Chrome/Edge/Safari 最新两个主版本。
4. 网络：本地回环或局域网访问。

### 2.4 假设与依赖
1. 假设用户具备基础浏览器操作能力。
2. 假设导入文件来源为官方导出，表头与格式满足规则。
3. OCR 依赖本地模型组件可用。
4. 当前版本默认无账号体系，部署在可信环境中使用。

## 3. 功能需求

## 3.1 FR-01 收支记录新增（P0）
功能描述：用户可新增单笔收入或支出记录。

前置条件：系统服务正常，分类列表可用。

主事件流：
1. 用户选择交易类型（收入/支出）。
2. 用户填写分类、金额、日期、备注。
3. 用户提交后，系统进行字段校验。
4. 校验通过后写入数据库并返回最新记录。

备选事件流：
1. 金额小于等于 0：提示“金额必须大于 0”。
2. 分类为空或超长：提示“分类不能为空且长度不超过 50”。
3. 备注超过 200 字：提示“备注长度不能超过 200”。

后置条件：新记录可在列表中查询，并参与统计计算。

验收标准：新增成功率达到 99.9%（非非法输入场景）。

## 3.2 FR-02 记录列表与筛选（P0，改进）
功能描述：支持按条件快速检索交易记录。

前置条件：数据库存在交易记录。

主事件流：
1. 用户进入记录列表页。
2. 用户可设置筛选条件：时间区间、类型、分类、金额区间、关键词。
3. 系统返回符合条件的数据，默认按“日期降序、ID 降序”显示。

备选事件流：
1. 筛选结果为空：展示空状态提示，不报错。
2. 条件非法（如最小金额大于最大金额）：提示并阻止查询。

后置条件：用户可定位目标记录，且筛选状态可重置。

验收标准：
1. 在 1 万条记录下，筛选响应时间不超过 1 秒。
2. 筛选结果与条件匹配准确率 100%。

## 3.3 FR-03 记录编辑与删除（P0，改进）
功能描述：支持修改和删除已有记录。

前置条件：目标记录存在。

主事件流：
1. 用户在列表中选择“编辑”或“删除”。
2. 编辑时系统展示原始值并允许保存。
3. 删除时系统二次确认。
4. 操作完成后刷新列表和图表。

备选事件流：
1. 记录不存在：返回“记录不存在”（404）。
2. 编辑保存校验失败：提示具体字段错误。

后置条件：数据库和统计图同步更新。

验收标准：编辑、删除后 1 秒内可见最新结果。

## 3.4 FR-04 OCR 票据识别与批量导入（P0）
功能描述：上传票据图片后生成候选交易，支持编辑后批量入库。

前置条件：上传文件为图片类型，OCR 组件可用。

主事件流：
1. 用户上传图片并点击识别。
2. 系统返回候选记录（含类型、分类、金额、日期、备注）。
3. 用户勾选并修正候选项。
4. 用户执行批量导入。

备选事件流：
1. 非图片或空文件：提示“仅支持图片文件/图片内容为空”。
2. OCR 失败：提示“识别失败”并保留页面可继续操作。

后置条件：被勾选且合法的候选项写入数据库。

验收标准：
1. OCR 预览接口返回时间不超过 8 秒（单张图片不大于 10MB）。
2. 单次最多返回 50 条候选记录。

## 3.5 FR-05 支付宝 CSV 导入（P0）
功能描述：解析支付宝 CSV 账单并导入系统。

前置条件：CSV 文件编码和表头符合要求。

主事件流：
1. 用户上传 CSV 并执行预览。
2. 系统按规则解析记录，生成候选项。
3. 用户编辑、勾选后提交导入。
4. 系统返回新增与跳过数量。

备选事件流：
1. 缺少必须表头：提示缺失项。
2. 编码不可识别：提示编码错误并给出可支持编码范围。
3. 文件为空：提示“CSV 文件内容为空”。

后置条件：合法且不重复的记录写入数据库。

验收标准：
1. 支持 gb2312、gb18030、gbk、utf-8-sig、utf-8。
2. 返回 inserted 与 skipped 两项统计。

## 3.6 FR-06 微信 XLSX 导入（P0）
功能描述：解析微信 XLSX 账单并导入系统。

前置条件：文件格式为 xlsx 且包含固定表头。

主事件流：
1. 用户上传 XLSX 并执行预览。
2. 系统解析第一工作表并识别有效账单行。
3. 用户编辑、勾选后提交导入。
4. 系统返回新增与跳过数量。

备选事件流：
1. 非 xlsx 文件：提示“仅支持微信账单 XLSX 文件”。
2. 未找到固定表头：提示文件不符合微信导出格式。
3. 状态非“支付成功/已收款”：自动跳过。

后置条件：合法且不重复的记录写入数据库。

验收标准：导入结果与源文件有效数据行一致，不重复落库。

## 3.7 FR-07 幂等去重导入（P0）
功能描述：同一账单重复导入时自动跳过已存在记录。

前置条件：导入项包含 source 与 import_key。

主事件流：
1. 系统按 source + import_key 判断是否重复。
2. 对重复项执行跳过，对新项执行写入。
3. 返回 inserted/skipped 统计。

备选事件流：
1. 唯一约束冲突：系统吞吐处理，不中断整批导入。

后置条件：数据库中同一来源同一键仅保留一条记录。

验收标准：重复导入同一文件 3 次，累计新增条数不变化。

## 3.8 FR-08 统计分析看板（P0）
功能描述：展示按分类统计和按月趋势统计。

前置条件：至少存在一条交易记录。

主事件流：
1. 系统加载支出分类统计。
2. 系统加载月度收支与结余统计。
3. 前端渲染图表并与最新数据同步。

备选事件流：
1. 无数据：显示空图状态。
2. 接口异常：提示加载失败，不阻塞其他模块。

后置条件：用户可按图表理解收支结构和变化趋势。

验收标准：图表与明细汇总数据一致，误差为 0。

## 3.9 FR-09 预算管理与超支预警（P1，新增）
功能描述：支持按月设置总预算和分类预算，并进行预警。

前置条件：用户已设置预算阈值。

主事件流：
1. 用户设置当月预算（总预算或分类预算）。
2. 系统实时计算已支出占比。
3. 达到阈值时显示预警（80%）和超支告警（100%）。

备选事件流：
1. 预算值非法（小于等于 0）：阻止保存并提示。
2. 未配置预算：不显示告警模块。

后置条件：用户可基于预算状态调整消费行为。

验收标准：预算占比计算误差不超过 0.01。

## 3.10 FR-10 数据备份与恢复（P1，新增）
功能描述：支持手动导出与导入备份文件。

前置条件：系统可访问数据库文件路径。

主事件流：
1. 用户点击“导出备份”，系统生成带时间戳的备份文件。
2. 用户可选择备份文件执行恢复。
3. 系统校验备份结构后执行恢复并提示结果。

备选事件流：
1. 备份文件损坏或格式不匹配：阻止恢复并提示。
2. 恢复失败：保留原数据并输出错误信息。

后置条件：恢复成功后系统数据与备份一致。

验收标准：
1. 备份成功率 100%。
2. 恢复成功后抽样校验关键数据一致率 100%。

## 3.11 FR-11 导入质量与异常提示标准化（P1，新增）
功能描述：统一 OCR、支付宝、微信三类导入链路的校验规则、错误码和提示文案。

前置条件：导入预览接口可正常访问。

主事件流：
1. 系统在导入预览阶段执行统一校验：文件格式、必填表头、必填字段、金额与日期格式、重复键冲突。
2. 系统为每条异常返回标准化结果：错误级别（error/warning）、错误码、中文提示、定位信息（行号/字段）。
3. 前端按统一结构展示异常，支持按“阻断类异常”和“可忽略类异常”分组。
4. 用户修正后可再次预览，直至满足导入条件。

备选事件流：
1. 文件整体不可解析：返回统一错误码并阻断导入。
2. 仅部分记录异常：允许展示可导入与不可导入明细，用户可选择仅导入合法项。

后置条件：导入链路错误提示一致、可定位、可复现。

验收标准：
1. 三类导入接口错误响应结构一致率 100%。
2. 异常提示包含错误码与中文提示，覆盖率 100%。
3. 对同一异常输入，返回结果稳定一致（幂等）且可复现。

## 4. 非功能需求

### 4.1 性能需求
1. 本地环境下首页初始化（列表+图表）耗时不超过 2 秒（1 万条记录以内）。
2. 单笔新增接口平均响应时间不超过 300ms。
3. 批量导入 2000 条记录时，后端处理时间不超过 8 秒。

### 4.2 安全性需求
1. 所有输入字段必须经过服务端校验（类型、长度、范围）。
2. 上传文件仅允许白名单格式（图片、CSV、XLSX），并校验空文件。
3. 错误响应不得暴露敏感堆栈与系统路径。
4. 数据库写入必须使用 ORM 或参数化语句，防止注入风险。

### 4.3 可用性需求
1. 所有操作错误需返回可理解的中文提示。
2. 关键高风险操作（删除、恢复）必须二次确认。
3. 导入流程需提供“预览-编辑-确认导入”闭环，避免误入库。
4. 页面在 1366x768 和移动端窄屏下均可用，无核心功能遮挡。

### 4.4 兼容性需求
1. 浏览器兼容 Chrome/Edge/Safari 最新两个主版本。
2. 后端兼容 Python 3.10 及以上版本。
3. 数据文件兼容 SQLite，支持后续迁移至 MySQL 的数据模型扩展。

## 5. 数据需求

### 5.1 交易记录字段
1. id：主键。
2. type：income 或 expense。
3. category：1-50 字符。
4. amount：大于 0。
5. note：可空，最大 200 字符。
6. txn_date：交易日期。
7. source：导入来源（可空）。
8. import_key：导入唯一键（可空）。
9. created_at：创建时间。

### 5.2 数据一致性规则
1. source + import_key 组合唯一。
2. 月度结余计算公式：balance = income - expense。
3. 删除记录后统计结果必须同步刷新。

## 6. 里程碑与交付范围

### 6.1 本期交付（V1.1）
1. P0：FR-01 至 FR-08 全部完成。
2. P1：FR-09、FR-10、FR-11 至少完成其一。

### 6.2 延期候选（V1.2+）
1. 多用户与权限体系。
2. 多币种与汇率支持。
3. 智能分类模型持续训练。

## 7. 验收标准汇总
1. 功能验收：所有 P0 功能通过联调与回归测试。
2. 性能验收：满足第 4.1 节指标。
3. 数据验收：导入去重、统计一致性、备份恢复验证通过。
4. 可用性验收：关键路径（新增、筛选、编辑、导入、统计）全流程可完成。
